# 預支款為什麼獨立成 CashAdvance

本專案把「預支款」做成 `CashAdvance`，不是只在報銷單上放一個文字欄位。原因是出納月底核帳時，需要知道公司先給了誰多少錢、哪些已經被核准報銷單核銷、還差多少沒有對上。

`ExpenseReport` 仍然是報銷單 aggregate root，明細只能透過報銷單新增或移除。預支款則是另一個 aggregate root；報銷單如果是預支費用，只保存 `CashAdvanceId` 這個 ID 參照，不直接持有或修改整個 `CashAdvance` 物件。

目前範圍刻意只做到「可追蹤、可核對、可查未結清」：

- 預支款記錄領款人、用途、日期、金額。
- 報銷單可選擇員工墊款或預支費用。
- 預支費用報銷單必須連到一筆預支款。
- 預支款列表用已核准報銷單加總核銷金額，顯示差額。
- 若核准報銷金額與預支金額不同，可在預支款底下新增實際結清紀錄：公司補付或員工繳回。

結清紀錄做成 `CashAdvance` aggregate 內部 entity，不單獨建立 repository。原因是它沒有獨立生命週期：它只是在說「這筆預支款後來實際補付 / 繳回多少」，所以新增紀錄一律走 `cashAdvance.AddSettlementRecord(...)`。

但「目前到底還要補付或繳回多少」不是 `CashAdvance` 自己能單獨知道的事，因為它需要同時查詢已核准的 `ExpenseReport`。因此本專案把差額與尚待結清金額放在 `CashAdvanceAppService` 組 DTO 時一起計算，Domain entity 只守住單筆結清紀錄的基本規則，例如結清金額必須大於 0、處理人不可空白。

後續補上修改 / 標記為不採用時，仍然維持同一個 aggregate 邊界：修改與不採用都透過 `CashAdvance` root 操作，不讓外層直接改 `CashAdvanceSettlementRecord`。不採用不是從資料庫刪除，而是保留處理人、原因與時間；列表核對金額只加總仍被採用的紀錄，詳情頁仍能看完整歷史。

預支款本身可修改用途與金額，但金額修改會影響差額與應結清金額，所以只允許在「尚未被任何報銷單引用、也沒有已計入核對的結清紀錄」時修改。結清紀錄是否已計入核對屬於 `CashAdvance` aggregate 內部規則；是否已有報銷單引用則是跨 aggregate 查詢，放在 `CashAdvanceAppService` 判斷。

同一筆預支款只要還有 Draft / Submitted / Returned 的關聯報銷單，系統只顯示暫估核對，暫不允許新增最終結清紀錄。這是為了避免流程中的報銷單後續核准後改變應補付 / 應繳回方向。

這裡仍然沒有做完整會計總帳，也刻意不支援沖銷、付款憑證或真正的出納日記帳。若之後要進入會計帳範圍，再另外設計，不把第一階段做得過重。
